# /team vs /ralph-loop 개발 방식 비교

## 1. 각 방식의 동작 원리

### `/team` (팀 가동 방식)

```
사용자 → /team "구현해줘" --mode step
    ↓
Orchestrator: 요청 분석 → 기술 스택 결정 → 작업 분해 → 전문가 배정
    ↓
Bootstrapper: 환경 구축
    ↓
전문가들 병렬 작업: Backend + Security(병렬 검토) + QA(TDD 가이드)
    ↓
QA + Security: 최종 검증
    ↓
완료 보고
```

**특징**: 한 번의 명령으로 분석 → 설계 → 구현 → 검증까지 **순차적으로 진행**.
각 Phase에서 역할별 전문가가 자동 활성화되어 검토를 수행합니다.

---

### `/ralph-loop` (반복 루프 방식)

```
사용자 → /ralph-loop "engine.py 구현해라" --max-iterations 10
    ↓
[반복 1] 프롬프트 수신 → 테스트 작성 → 구현 시도
    ↓
[반복 2] 같은 프롬프트 → 이전 코드 확인 → 테스트 실행 → 실패 수정
    ↓
[반복 3] 같은 프롬프트 → 테스트 확인 → 추가 수정 → 통과
    ↓
<promise>DONE</promise> → 루프 종료
```

**특징**: 같은 프롬프트를 반복 주입하여, Claude가 자신의 **이전 작업 결과를 보면서 점진적으로 개선**.

---

## 2. 핵심 차이점

| 비교 항목 | `/team` | `/ralph-loop` |
|-----------|---------|---------------|
| **실행 구조** | 단일 패스 (1회 실행) | 반복 루프 (N회 실행) |
| **역할 분리** | 10개 전문가 역할 자동 활성화 | 역할 구분 없음 (일반 Claude) |
| **작업 범위** | 넓음 (Phase 단위 가능) | 좁음 (단일 모듈 권장) |
| **보안 검토** | Security가 자동 개입 | 프롬프트에 명시해야 함 |
| **TDD 가이드** | QA가 테스트 선행 작성 유도 | 프롬프트에 TDD 명시 필요 |
| **자기 교정** | 1회 실행이므로 제한적 | **핵심 강점** — 반복하며 자기 수정 |
| **완료 판단** | Orchestrator 체크리스트 | `<promise>` 태그 또는 max-iterations |
| **비용 (토큰)** | 1회 실행 비용 | 반복 횟수 × 프롬프트 비용 |

## 3. 장단점 비교

### `/team`의 장점

| # | 장점 | 설명 |
|---|------|------|
| 1 | **전문가 역할 분리** | Security 보안 검토, QA 테스트 전략 등 역할별 관점이 자동 반영 |
| 2 | **구조적 접근** | Orchestrator가 작업 분해, 의존성 파악, 체계적 진행 |
| 3 | **보안 자동 개입** | 별도 요청 없이 Security Engineer가 각 단계에서 검토 |
| 4 | **설계 일관성** | 하나의 실행 흐름에서 모듈 간 인터페이스 일관성 유지 |
| 5 | **넓은 범위 처리** | Phase 단위(여러 모듈)를 한 번에 처리 가능 |

### `/team`의 단점

| # | 단점 | 설명 |
|---|------|------|
| 1 | **자기 교정 부재** | 1회 실행이므로 테스트 실패 시 스스로 반복 수정하지 않음 |
| 2 | **컨텍스트 포화** | 범위가 너무 크면 후반부 품질 저하 가능 |
| 3 | **런타임 버그 취약** | 코드를 실행해봐야 발견되는 버그에 대응 어려움 |

### `/ralph-loop`의 장점

| # | 장점 | 설명 |
|---|------|------|
| 1 | **자기 교정** | 테스트 실패 → 다음 반복에서 자동 수정. **최대 강점** |
| 2 | **점진적 개선** | 매 반복마다 이전 결과를 보고 개선, 품질이 수렴 |
| 3 | **명확한 완료 기준** | 테스트 통과 = 완료. 객관적이고 검증 가능 |
| 4 | **런타임 버그 강건** | 실행 → 실패 → 수정 → 재실행 루프가 자연스러움 |

### `/ralph-loop`의 단점

| # | 단점 | 설명 |
|---|------|------|
| 1 | **역할 전문성 없음** | Security, Performance 등 전문가 관점이 자동 적용되지 않음 |
| 2 | **좁은 범위만 효과적** | 여러 모듈을 한 번에 넣으면 비효율적 |
| 3 | **비용 증가** | 반복 횟수만큼 토큰 소비 |
| 4 | **프롬프트 품질 의존** | 프롬프트가 모호하면 반복해도 수렴하지 않음 |

## 4. 이 프로젝트(korean-quant) 적용 분석

### 프로젝트 특성

이 프로젝트는 **문서화가 비정상적으로 잘 되어 있습니다**:
- docs/01~08: 모듈별 메서드 시그니처, 코드 템플릿, 엣지 케이스까지 기술
- docs/CLAUDE.md: 알려진 이슈 5개, 코딩 컨벤션 명시
- docs/08_checklist.md: 버그 목록 11개 사전 정리

이런 프로젝트에서는 `/team`의 1회 실행으로 대부분의 모듈을 올바르게 생성할 수 있습니다.

### Phase별 평가

| Phase | 모호함 | 런타임 버그 가능성 | `/team` 완료 확률 | `/ralph-loop` 필요? |
|-------|--------|-------------------|-------------------|---------------------|
| 1. 환경 구축 | 거의 없음 | 없음 | **95%** | 불필요 |
| 2. 데이터 파이프라인 | 낮음 | pykrx 응답 형식 | **80%** | 낮음 (문서에 해결책 있음) |
| 3. 팩터 구현 | 낮음 | 엣지 케이스 | **85%** | 낮음 (공식 명확) |
| **4. 백테스트** | **중간** | **높음** | **70%** | **높음** |
| 5. 키움 API | 낮음 | mock 테스트 | **85%** | 낮음 (스펙 명확) |
| 6. 자동화 | 낮음 | 통합 이슈 | **85%** | 낮음 (정형 패턴) |

### Phase 4가 `/ralph-loop`이 필요한 이유

백테스트는 **코드를 실행해봐야 발견되는 버그**가 집중된 영역입니다:

```
문서에 정의된 버그 11개 중 4개가 백테스트/스케줄러 관련:
- pd.offsets.BMonthEnd() 특정 월 동작
- T일 신호 → T+1 체결 날짜 정렬
- 거래 비용 방향별 적용 (매도세 vs 매수수수료)
- freq="BME" deprecated 문제
```

이런 버그는 코드 리뷰로 발견하기 어렵고, **실행 → 실패 → 수정**의 반복이 필요합니다. 이것이 `/ralph-loop`의 자기 교정이 빛나는 지점입니다.

### Phase 2~3이 `/team`으로 되는 이유

```
Phase 2 (데이터):
  - pykrx 한글 컬럼명 → 문서에 매핑 딕셔너리 이미 정의
  - 모멘텀 iloc 하드코딩 → 문서에 relativedelta 해결책 명시
  → 문서를 읽으면 1회에 올바른 코드 생성 가능

Phase 3 (팩터):
  - PBR 역수 변환, ROE 클리핑 범위 → 수학 공식이 docs/04에 명시
  - 자본잠식 제외 조건 → 명확한 if 문
  → 모호할 것이 없으므로 1회 실행 충분
```

## 5. 권장 워크플로우

```
/team (Phase 1~3)  →  /ralph-loop (Phase 4)  →  /team (Phase 5~6 + 최종)
    설계 + 구현            반복 교정                 통합 + 검증
```

| Phase | 명령 | 방식 | 이유 |
|-------|------|------|------|
| 1. 환경 구축 | `/team --mode step` | 1회 | 파일 생성, 반복 불필요 |
| 2. 데이터 | `/team --mode step` | 1회 | docs/03 스펙 상세 |
| 3. 팩터 | `/team --mode step` | 1회 | 수학 공식 명확 |
| **4. 백테스트** | **`/ralph-loop --max-iterations 10`** | **반복** | **런타임 버그 다수** |
| 5. 키움 API | `/team --with backend,security,qa` | 1회 | API 스펙 명확 |
| 6. 자동화 | `/team --with backend,qa` | 1회 | 정형 패턴 |
| 최종 검증 | `/team --with qa,security` | 1회 | 전문가 관점 |

### 유연한 대응

Phase 2~3에서 예상외로 테스트 실패가 반복되면, 해당 모듈만 `/ralph-loop`으로 전환할 수 있습니다. 하지만 이 프로젝트의 문서 품질을 고려하면 그럴 가능성은 낮습니다.

## 6. 시나리오별 추천 (일반)

| 시나리오 | 추천 | 이유 |
|----------|------|------|
| 문서가 잘 된 프로젝트 | `/team` 중심 | 1회 실행으로 충분 |
| 스펙이 모호한 프로젝트 | `/ralph-loop` 중심 | 반복 탐색이 필요 |
| 런타임 디버깅 필요 | `/ralph-loop` | 실행 → 수정 반복 |
| 보안 감사 | `/team --with security` | 전문가 관점 필수 |
| 성능 최적화 | `/team --with performance` | 전문가 분석 필요 |
| 단일 버그 수정 | `/team --with backend,qa` | 1회로 해결 가능 |

## 7. 결론

### `/team`과 `/ralph-loop`은 **대체재가 아닌 보완재**

- **`/team`** = **아키텍트** (설계, 전문가 검토, 넓은 범위)
- **`/ralph-loop`** = **디버거** (실행, 실패 수정, 좁고 깊은 집중)

### 이 프로젝트에서의 현실적 결론

**문서 품질이 높으면 `/team`만으로 대부분 완료됩니다.**
`/ralph-loop`은 "실행해봐야 아는" 백테스트 같은 영역에 선택적으로 투입하는 것이 효율적입니다.

모든 모듈을 `/ralph-loop`으로 나누는 것은 과한 접근이며,
반대로 전체를 `/team` 한 번에 넣는 것도 컨텍스트 포화 위험이 있습니다.
**Phase 단위 `/team` + 백테스트만 `/ralph-loop`**이 이 프로젝트의 최적해입니다.
