# 운용 Runbook

> 한국 멀티팩터 퀀트 시스템의 일상 운용 루틴. 분기 리밸런싱 전제.
> 백테스트로 검증된 alpha를 보호하는 것이 최우선 목표.

---

## 일간 (5분, 매일 장 종료 후)

### 확인 사항
- [ ] 텔레그램 알림 정상 도착 확인 (오늘 일간 리포트, 리스크 감시 알림)
- [ ] `risk_guard` 알림 검토
  - 종목별 손절 경고 (기본 -20%): 종목명·하락률
  - 포트폴리오 드로다운 경고 (기본 -15%)
  - 관리종목 지정 감지
  - 폐지 임박 dry_run 알림: 종목명·카테고리·days_until 확인
  - `dry_run=false` 후 자동매도 발생 시: 체결 확인
- [ ] 에러 로그 확인 (`logs/quant.log` tail)
- [ ] 키움 API 토큰 만료 여부 (24시간 유효, 자동 갱신 확인)

### 하지 않는 것
- 개별 종목 가격 확인 (시스템이 분기 단위로 관리)
- 임의 매수/매도 (시스템 alpha 파괴)
- 시장 뉴스 보고 즉흥 결정

---

## 주간 (15분, 매주 금요일 장 종료 후)

### 확인 사항
- [ ] 주간 누적 손익 확인 (대시보드 또는 키움 잔고)
- [ ] `risk_guard` 주간 알림 누적 검토
- [ ] `dart_notifier` 공시 알림 중 보유 종목 관련 건 검토
  - 합병/분할, 최대주주 변경, 자본 변동 등은 분기 리밸런싱에서 반영
  - 즉시 매도가 필요한 경우는 거의 없음 (관리종목 지정만 예외)
- [ ] DART 공시 알림 누적량 확인 (api_limit.daily_warning_threshold 도달 여부)

### 하지 않는 것
- 종목 교체 (다음 리밸런싱까지 대기)
- 벤치마크 대비 초조해하기 (분기 단위 평가)
- 시스템 파라미터 변경

---

## 월간 (30분, 매월 마지막 영업일)

### 확인 사항
- [ ] 월간 수익률 vs KOSPI 지수 비교
- [ ] 섹터 노출 현황 확인 (Sanity Report 누적: `docs/reports/sanity_*.md`)
- [ ] 보유 종목 중 다음 분기 리밸런싱에서 교체될 가능성 높은 종목 파악
  - `screener.screen()` 재실행 (매매 없이 확인만)
  - composite score 순위 변화 관찰
- [ ] `risk_guard` dry_run 1개월 누적 분석
  - 폐지 임박 dry_run 알림이 안정적으로 작동하면 `dry_run=false`로 전환 검토
- [ ] 텔레그램/DART 채널 헬스체크 (`/start`, `/status` 명령)

### 하지 않는 것
- 시스템 파라미터 변경 (분기 검토 시점에만)
- 임의 종목 추가/제거
- 백테스트 재실행 후 즉흥 튜닝

---

## 분기 (반나절, 리밸런싱 날)

### 리밸런싱 전 (D-1 또는 당일 장 시작 전)
- [ ] Sanity Report 확인 (자동 생성: `docs/reports/sanity_{date}.md`)
  - 주의 종목 (⚠️ 플래그) 검토
  - 섹터 분포 확인 (HHI, 최대 섹터)
  - 신규 편입/퇴출 종목 파악
  - 부채비율 > 300% 또는 영업이익 음수 종목 수동 확인
- [ ] 시장 레짐 확인 (KOSPI MA200 위/아래, market_regime 투자 비중)
- [ ] 가용 현금 확인 (`/balance` 키움 잔고)
- [ ] DART 공시 — 최근 2주간 보유 종목 + 후보 종목 중대 공시 확인

### 리밸런싱 실행
- [ ] 시스템 자동 실행 대기 (scheduler, 08:50 트리거 → T+1 09:00 시가 체결)
- [ ] 체결 알림 확인 (텔레그램)
- [ ] 미체결 건 확인 + 수동 처리 (있으면)
  - 거래정지·상한가/하한가·유동성 부족으로 미체결 가능
  - 다음 영업일 시가에 재시도 또는 스킵 판단

### 리밸런싱 후
- [ ] 실거래 체결가 vs 백테스트 시뮬 체결가 비교 (있으면)
- [ ] 분기 성과 리포트 작성/확인
  - 실거래 수익률 vs 백테스트 예상 수익률
  - 괴리 > 2%p이면 원인 분석 (슬리피지? 시장충격? 미체결? 호가단위?)
- [ ] POLICY 검토: 시스템 변경 필요 여부 판단 (`docs/POLICY.md`)
  - 시스템 변경 5조건 통과 시에만 진행
- [ ] 다음 분기 리밸런싱 날짜 캘린더 등록

## 비정기 이벤트 대응

### 관리종목 지정 알림 수신 시
1. `risk_guard` 알림 내용 확인 (종목명, 카테고리, days_until)
2. `delisting_auto_sell_enabled=true` + `dry_run=false`면 자동 매도 대기
3. `dry_run=true`면 수동 매도 판단
   - failure / expired / other 카테고리만 자동 매도 대상
   - merger / voluntary 카테고리는 절대 자동매도 금지 (보상 받을 수 있음)
4. 매도 후 현금은 다음 리밸런싱까지 보유 (중간 재투자 금지)

### KOSPI -5% 이상 급락 시
1. **패닉 매도 금지** — 시스템이 시장 레짐으로 다음 리밸런싱에 비중 조절
2. `risk_guard` MDD 알림 확인 (드로다운 경고)
3. 월간 검토를 앞당겨 실행 (손실 규모 파악)
4. **시스템 끄지 말 것** — 가장 흔한 실패 패턴
5. 분기 리밸런싱은 정상 스케줄대로 실행

### KOSPI MA200 하향 돌파 (약세장 진입)
1. `market_regime`이 자동으로 투자 비중 축소 (defensive_ratio = 30%)
2. 다음 리밸런싱에서 자동 반영
3. 운용자 추가 조치 불필요

### 시스템 에러/장애 시
1. `logs/quant.log` 확인
2. DB 백업 확인 (`data/quant.db`, `data/monitor.db`)
3. exe 재시작 (또는 `python -m gui`)
4. 키움 API 토큰 만료 시 재인증
5. 스케줄러 Job 누락 시 수동 트리거 (`scheduler/main.py` 참조)
6. 텔레그램·DART API 키 만료 확인

### 백테스트 결과 의심 시
1. 코드 변경 전 항상 백테스트 재실행 후 비교 (baseline 보존)
2. CAGR 차이 ±0.5%p 이내면 노이즈로 간주
3. 큰 변화면 git diff로 원인 추적
4. `docs/POLICY.md` 5조건 통과 후에만 채택

## 절대 금지 사항

1. **분기 사이 임의 매수/매도** — 백테스트 alpha 파괴 (8년 평균 12% Alpha 검증)
2. **"이번만" 예외 적용** — 한 번 시작하면 90%가 됨
3. **타인 추천 종목 편입** — 시스템 밖 종목은 검증 불가
4. **MDD 공포에 시스템 OFF** — 가장 흔한 실패 패턴 (역검증: 최악 구간 -0.74% Alpha)
5. **백테스트 재실행 후 즉흥 파라미터 변경** — `docs/POLICY.md` 5조건 통과 없이 변경 금지
6. **`null` 비활성화 우회** — `max_drawdown_pct: 0.99` 같은 패턴 사용 금지
7. **dev/prod 스키마 차이 무시** — DB 스키마 변경은 추가만 허용 (`docs/DB_SCHEMA_POLICY.md`)
8. **API 키 git 커밋** — `.env`만 사용, 코드 하드코딩 금지
